# Phoenix RAG Baseline 1: answer completeness + retrieval relevance

Date: 2026-03-25

## Goal

Run the first end-to-end Phoenix evaluation pass for the current local RAG setup and extract actionable retrieval signals.

## Setup

- Generation model: `qwen2.5:1.5b-instruct-q4_K_M`
- Judge model: `qwen2.5:7b-instruct-q4_K_M` via Ollama
- Trace transport: Phoenix / OpenInference spans
- Corpus chunking strategy: structural chunking
- Important corpus property:
  structural chunks correspond to sections and subsections, so each chunk already carries concentrated meaning

## Evaluation scripts

- [`answer_completeness.py`](/home/olesia/code/prompt_gen_proj/Measurement/Evaluation/Phoenix/answer_completeness.py)
  - judges final answer completeness
  - writes `answer_completeness` annotations to `rag.request`
- [`retrieval_relevance.py`](/home/olesia/code/prompt_gen_proj/Measurement/Evaluation/Phoenix/retrieval_relevance.py)
  - judges each retrieved chunk as `relevant`, `partial`, or `irrelevant`
  - writes:
    - `retrieval_relevance_mean_topk`
    - `retrieval_relevance_weighted_topk`
  - target span: `retrieval.vector_search`

## Answer Completeness Result

Observed result in Phoenix UI:

- `answer_completeness` annotations were successfully written to `rag.request`
- trace-level aggregate in Phoenix UI was approximately `mu = 0.72`

Interpretation:

- this is a workable local baseline for a small generation model
- the pipeline is healthy enough to produce useful answers on the current benchmark set
- the score is good enough to justify deeper retrieval analysis before replacing the generation model

## What The First Experiment Already Tells Us

The first experiment already paid off.

The main pattern is consistent:

- rank 1 is usually useful
- rank 2 is often still useful or partially useful
- rank 3 and rank 4 are frequently noise

This strongly suggests that the current system is retrieving more context than the generator really needs.

## Hypothesis For The Next Experiment

Likely good compromise:

- add a reranker
- pass only `top2` chunks to the chat model

Why this is plausible:

- this corpus uses structural chunking
- chunks represent sections or subsections
- semantic content is therefore dense
- once the first one or two chunks are good, additional chunks often add less signal than noise

## Recommended Next Step

Run the next controlled experiment:

- keep the same question set
- compare `top4` vs `top2`
- measure:
  - `answer_completeness`
  - `retrieval_relevance_mean_topk`
  - `retrieval_relevance_weighted_topk`
  - latency

If `answer_completeness` stays stable while latency and retrieval purity improve, then `top2` becomes the new baseline and reranking becomes the next high-value upgrade.
